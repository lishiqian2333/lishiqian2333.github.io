UniswapV3 - 原理

优势

Uniswap V2具有灵活，安全，高效的优点，引入了路由，价格预言机等技术，是DeFi领域的龙头项目。
但是Uniswap V2存在资金利用率太低的问题，也就是说，大部分流动性提供者提供的流动性成为了交换池中的分母，实际上不能参与交换。
为了解决这个问题，MakerDao推出了Uniswap V3，它具有如下特点：

  1. 流动性的提供者可以选择任意的加个区间来提供流动性
  2. 针对不同的交易对，具有不同的费率设置
  3. 具有更先进的价格预言机
  4. LP不再为ERC20，而是ERC721.

虚拟流动性

在Uniswap V2的交易池中，代币的价格一般会在一个价格区间内波动。这说明在x * y = k 的AMM做市曲线中，曲线上大部分的点都不会被触及，池中的流动性代币大部分将会成为价格计算的分母，而不能真正被利用
Uniswap V3为了解决这一问题，允许用户在一定的价格区间提供流动性，而虚拟流动性用于抬升曲线，只在价格计算时被使用，而不会真正参与交易，这使得流动性提供者的资金能够被充分利用
例如：用户选择在[Price a, Price b]的区间提供流动性，那么在[Price a, Price b]中间的任意点，都有：
X = X real + X virtual
Y = Y real + Y virtual
其中，X real和Y real表示用户提供的真实的流动性数量，X virtual和Y virtual则是虚拟的x token和y token的数量
当价格处于用户设置的Price a时用户提供的流动性将会全部变成X：Y real = 0 | Y = Y virtual
同理，将价格处于用户设置的Price b时用户提供的流动性将会全部变成Y： X real = 0 | X = X virtual
当价格超出[Price a, Price b]范围，用户提供的流动性将会被暂时移除，以保证虚拟流动性不变，直到价格恢复进入[Price a, Price b]区间，用户提供的流动性被添加。

流动性聚合

Uniswap V3可以让用户指定特定的价格区间来提供流动性，这将意味着，一个池子中可能有很多不同的流动性。流动性聚合就是让用户的交易拆分，在不同的流动性中交易，最后将流动性聚合。而在Uniswap V3中，在交易前，需要保证各个流动性中的价格是一致的，而在交易后，也需要保证各个流动性的价格是一致的。因此，Uniswap V3围绕价格进行。
在Uniswap V2中，对于每个价格，流动性大小是相同的，而在Uniswap V3中各个对于每个价格，流动性大小将会发生变化。因此，在Uniswap V3中，交易的结果将会围绕价格进行。
也就是说，设置交易后价格下降到P1,流动性k = x * y不变。那么对于交换需要的x1,和获得的y1，可以推导出它们关于P1和k的表达式。
在实际交易中，k将会是P1价格区间的总流动性。而当交易价格跨越了流动性边界，那么将会在当前总流动性上增加或移除相应流动性。
当交易价格到达一个价格时，计算amountIn是否被耗尽，如果没有被耗尽，将amountOut增加后，剩下的amountIn将会进入下一个价格，继续交换amountOut，而如果价格跨越了流动性边界，则会在当前总流动性上增加或移除相应流动性，如此循环；直到当交易价格到达下一个价格时，amountIn被耗尽，结束循环，再计算添加amountIn耗尽时换出的amountOut。

关于交易价格

v3可以让用户指定交易的价格，v3需要存储这些价格。在价格轴上，有无穷的价格点，如果v3采用连续的价格，那么价格的记录将是无限的。
因此，价格是不连续的，他是一个个离散的点组成的集合，这些点用ticks表示。
这样，但是相邻价格点的间隔不能过大，否则就会造成精度问题。如过将间隔定位0.0001，使用等比，或等差排列价格，那么就会发现，uint256类型的整型是不够表示足够大的价格范围的。
我们会发现，在价格大的时候，我们对价格精度的需求会变低，而在价格小的时候，我们对精度的需求会变高。例如，当价格为10000000时，我们不会在意0.0001的误差。
因此，v3使用等幂数列表示价格离散： P i = （0.0001）^ i
其中i是一个有符号数，当i = 0时，p为1。 也就是价格的起始点。当i为正数时，当i越来越大，价格越来越大，价格间隔就会越来越大。当i为负数时，p i的范围为（0，1）

关于交易费率和tickspacing

相比v2固定的交易费率，v3设置了灵活的交易费率，初始有0.005%，0.3%，1%，并且可以由社区添加新的费率。
为什么要这样设计呢？由于对于一些价格波动比较小的交易对，交易产生的差价不大，例如稳定币的交易对，如果收取0.3%的交易费，不利于市场流动。
而一些山寨币的价格波动非常大，就算设置1%的交易费，也会有人愿意交易。
tickspacing的作用是在计算价格时，跳过tickspacing数量tick的价格计算，它的设置会进一步调整价格点的离散程度，并且节省交易者的计算gas
对于价格波动小的交易对，对价格精度需求大，tickspacing设置一般较小，对于价格波动大的交易对，设置大的tickspacing，进一步减少了gas消耗

关于手续费

v3中的手续费计算专注于单位流动性手续费量，而不是关注于各个价格流动性占总流动性的比率。
feeGrowthGlobal用来储存单位流动性手续费的累积值
feeGrowthGlobal += step fee / liquidity
其中step fee是每一个价格交易产生的手续费，liquidity是在这个价格内，有效的总流动性大小（我们知道，在不同价格中，有效的流动性大小可能不一样，因为在跨越ticks时，可能有一部分未激发的流动性被激发，从而添加到总流动性中，有一部分流动性被从总流动性中移除，因为超越了用户设定的价格区间）。
刚刚我们计算的时在（0，+∞）区间累积的手续费，由于不同的价格区间对手续费的贡献不同，那么如何将它们公平的分配给每一个价格区间呢？
我们在当前价格消耗流动性，产生手续费，应该将当前手续费累累积在这个tick里，但是这个价格区间外的累积的手续费不会变化。那么我们就在tick中储存在这个价格区间之外比其低的和高的区间的手续费
比如在区间（a，b），记录了（0，a）和（b，+∞）的手续费feeGrowthOutside below和feeGrowthOutside above​，那么我们可以使用feeGrowthGlobal - feeGrowthOutside below -feeGrowthOutside above得出本区间累积的手续费feeGrowthInside，这样就无需遍历ticks
得到了feeGrowthInside，乘上用户头寸中的流动性就是累积的手续费
